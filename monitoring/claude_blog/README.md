# Claude Blog Monitor

This workspace has a Codex cron automation that checks `https://claude.com/blog`
every 3 hours for newly published articles.

## Detection Contract

- Use `monitoring/claude_blog/seen_articles.json` as the durable state file.
- On each run, open `https://claude.com/blog` and extract visible article
  cards with canonical URL, title, publication date, and category when present.
- A post is considered new when its canonical URL is not in
  `seen_article_urls` and it is published after the baseline cutoff in the
  state file.
- Do not generate reports for posts that existed before the baseline unless
  the user explicitly asks for a backfill.
- Update `last_checked_at` after every successful check.
- Add each article URL to `seen_article_urls` only after both the Markdown and
  HTML reports have been written.

## Report Output

For each new article, create both files:

```text
monthly_report/YYYY-MM/daily_report/YYYY-MM-DD/YYYY-MM-DD_<slug>.md
monthly_report/YYYY-MM/daily_report/YYYY-MM-DD/YYYY-MM-DD_<slug>.html
```

The report should include:

- Source metadata: title, URL, publication date, detection timestamp.
- Overall summary.
- Technical innovation analysis.
- Enterprise scenario reference analysis.
- Practical implications, risks, and follow-up actions.

HTML reports should be standalone UTF-8 documents with readable inline CSS and
the same substantive content as the Markdown report.

## Backfill

Historical backfills are generated by:

```sh
node monitoring/claude_blog/backfill_2025_reports.mjs
```

The 2025-to-present backfill writes:

- `monitoring/claude_blog/backfill_2025_index.json`
- `monthly_report/claude_blog_2025_backfill_index.md`
- Per-article Markdown and HTML files under the normal monthly/daily archive.

After a successful backfill, `seen_articles.json` is updated with the article
URLs and generated report paths so the recurring monitor does not duplicate
historical reports.
